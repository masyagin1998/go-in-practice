# 06_sockets_zc — TCP + MSG_ZEROCOPY

То же, что `05_sockets` (loopback-TCP, Go — сервер, peer — клиент), но
peer отправляет через `send(fd, buf, len, MSG_ZEROCOPY)` — ядро пинит
user-страницы и кладёт их в TX-очередь TCP как scatter-gather, без
копирования в `skb`.

## API в трёх шагах

1. **Включить на сокете**: `setsockopt(fd, SOL_SOCKET, SO_ZEROCOPY, &on, ...)`.
2. **Отправить**: `send(fd, buf, len, MSG_ZEROCOPY)`. После этого нельзя
   модифицировать `buf`, пока ядро не подтвердит завершение.
3. **Дренировать error-queue**: `recvmsg(fd, &msg, MSG_ERRQUEUE)` читает
   completion-уведомления. Без этого очередь переполнится и `send`
   начнёт возвращать `ENOBUFS`.

## Когда реально выгодно

Требование: Linux kernel ≥ 4.14, **буфер ≥ ~16 KB**. Для мелких
сообщений ядро всё равно копирует (эвристика в коде сетевого стека —
pinning пары страниц дороже, чем memcpy 256 байт).

Типичный выигрыш в бенчмарке при 64 KB per message — до 30% throughput
на сервере, который льёт много одинаковых данных. Для пинг-понга с
мелкими сообщениями разница обычно в пределах шума, а оверхед на
управление error-queue может даже съесть выигрыш.

## Go-сторона

Go-стандартная библиотека `MSG_ZEROCOPY` напрямую не поддерживает —
нужен `syscall.Sendmsg` с ручным флагом и парсингом error-queue. Для
демо проще: zero-copy только на TX-клиенте (peer), Go-сервер читает и
отвечает через обычный `Write`.
