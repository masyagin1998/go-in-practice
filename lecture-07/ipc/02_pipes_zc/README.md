# 02_pipes_zc — pipe + vmsplice(SPLICE_F_GIFT)

То же, что `01_pipes`, только peer пишет в pipe через `vmsplice(2)` вместо
`write(2)`. Флаг `SPLICE_F_GIFT` говорит ядру: "забирай мои user-страницы
как есть, копировать не нужно".

## Чем отличается от обычного write

```
write():    user buf  ──copy──►  pipe buffer (kernel)  ──copy──►  reader
vmsplice:   user buf  ─ref────►  pipe buffer (kernel)  ──copy──►  reader
```

Сокращение с двух копий до одной. Вторую копию (pipe→reader) устранить
через splice можно, только если reader тоже fd пайпа/сокета — из
Go-стороны без cgo это неудобно, оставляем обычный read.

## Когда реально работает

- Write-end должен быть pipe или FIFO.
- Страницы должны быть **page-aligned и page-sized** (у нас
  `aligned_alloc(PAGESIZE, PAGESIZE)` на каждое сообщение).
- Для мелких сообщений (меньше страницы) ядро всё равно копирует —
  выигрыш виден на килобайтных/мегабайтных буферах.

## Ограничение

После `vmsplice(SPLICE_F_GIFT)` **нельзя модифицировать** отданную
страницу до того, как reader её прочитает — иначе reader увидит ваши
последующие записи. В этом демо мы выделяем отдельную страницу на
каждую итерацию, чтобы не переиспользовать.
