# sighup-reload

Классическая схема: процесс читает конфиг при старте, по `SIGHUP`
перечитывает и атомарно подменяет указатель. Без рестарта процесса,
без потери соединений/состояния.

Так работают nginx, postgres (`pg_reload_conf()` тоже сводится к SIGHUP
postmaster'у), sshd, haproxy, rsyslog.

## Ручной сценарий

```bash
go run . &
cat /tmp/sighup_demo.conf
echo '{"log_level":"debug","workers":16}' > /tmp/sighup_demo.conf
kill -HUP $!
```

Видно: `[tick]` начинает печатать новые значения сразу после `SIGHUP`.

## Ключевые детали

- **`atomic.Pointer[config]`** — подмена указателя на конфиг потокобезопасна
  без мьютекса. Читатели могут держать старый `*config` сколько угодно —
  главное, что новый `Load()` увидит обновление.
- **`signal.Notify` канал буферизован** (ёмкость 1). Если два SIGHUP
  придут близко друг к другу, второй *может* потеряться — это ОК: всё
  равно перечитываем актуальный файл, результат тот же.
- **Атомарная запись файла**. В реальности конфиг лучше писать через
  `tmpfile + rename` (атомарная подмена на уровне VFS), а SIGHUP слать
  уже после. В этом демо упрощено до `os.WriteFile`.
- **Ошибки валидации**. Плохой JSON / невалидные значения — **не
  должны** убивать процесс. Логируем и продолжаем работать со старым
  конфигом (здесь именно так: при ошибке `Load()` не меняет `current`).
