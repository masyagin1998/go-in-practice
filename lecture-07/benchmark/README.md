# benchmark — сравнение IPC-транспортов (+ zero-copy)

Ping-pong round-trip между Go-клиентом и C-peer'ом. Для каждого
транспорта гоняются **два размера сообщения**: 256 B (где zero-copy
обычно проигрывает из-за оверхеда) и 65 536 B (где zc-варианты
реально выигрывают).

Имена транспортов совпадают с `../ipc/01_..11_` для сквозной навигации.

## Запуск

```bash
# Полный дефолт (~7 минут на i9): REPEATS=20, ITERS=100_000
make bench

# Короткий прогон (~40 секунд): REPEATS=3, ITERS=50_000
make bench-quick

# Произвольно:
BENCH_REPEATS=10 BENCH_ITERS=200000 make bench
```

Перед прогоном Makefile собирает C-пиры: `peer_pipe`, `peer_pipe_zc`,
`peer_fifo`, `peer_fifo_zc`, `peer_tcp`, `peer_tcp_zc`, `peer_uds`,
`peer_uds_zc`, `peer_mq`, `peer_shmsem`.

## Что видно по числам

### На маленьких сообщениях (256 B) zero-copy не помогает

- `02_pipe_zc` ≈ `03_fifo` ≈ `04_fifo_zc` ≈ `01_pipe` — все в пределах
  5–15% друг от друга. `vmsplice(SPLICE_F_GIFT)` для 256-байтного
  буфера меньше одной страницы — ядро всё равно копирует, а API-оверхед
  чуть-чуть съедает выигрыш.
- `06_tcp_zc` слегка **медленнее** `05_tcp`: `MSG_ZEROCOPY` на мелочи
  бесполезен, зато дренирование error-queue добавляет пару syscall'ов.
- `08_uds_zc` медленнее `07_uds`: плата за `mmap` + memfd setup
  (one-time, но на 256 байт per rtt всё равно заметно).
- `10_shmsem` выигрывает всегда: futex wake/wait — два syscall'а на
  round-trip, данные только через cache-line traffic.

### На больших сообщениях (64 KB) zero-copy начинает работать

- `10_shmsem` ведёт с отрывом в ~1.3× от следующего: данные не
  копируются вообще, только `memcpy` в peer'е для честного echo.
- **`08_uds_zc`** на втором месте с **4.76 GB/sec** — почти на уровне
  shmem, но через fd-passing вместо named-объекта. Когда отключаем
  копии, UDS догоняет shmem.
- **`04_fifo_zc`** и **`02_pipe_zc`** дают +20–25% к обычным
  `fifo`/`pipe`: `vmsplice(SPLICE_F_GIFT)` реально экономит копию
  user→kernel, раз буфер page-aligned.
- **`06_tcp_zc` всё ещё медленнее `05_tcp`** — на loopback нет
  реальной NIC-передачи, выигрыш от `MSG_ZEROCOPY` поглощается
  overhead'ом pinning и управлением error-queue. На реальной сети с
  большим BDP картина обратная.

## Методология

- **Warmup**: перед измерениями один прогон выкидываем. Без него
  первый run всегда медленнее из-за холодных кешей, lazy page-faults,
  lazy page-table population.
- **Медиана, не среднее**: выбросы (scheduler увёл на busy CPU,
  GC-пауза, preemption) портят mean. Median устойчива.
- **TCP на port=0**: каждый run получает свежий ephemeral-порт, иначе
  `TIME_WAIT` от предыдущего заставил бы bind ждать 60 сек.
- **Честное копирование в shm+sem**: peer делает `memcpy` в локальный
  буфер и обратно — иначе замер измерял бы только `sem_wait/post`.
- **`TCP_NODELAY`** включён с обеих сторон (без этого Nagle добавил бы
  ~40 мс на каждый round-trip).

## Почему mq пропускается на 64 KB

`/proc/sys/fs/mqueue/msgsize_max` по умолчанию = **8192**. `mq_open`
с большим `mq_msgsize` возвращает `EINVAL`. Без `CAP_SYS_RESOURCE` /
`root` лимит не поднять — поэтому бенч честно пропускает mq.

## Ограничения (не продакшн-бенч)

- CPU governor, NUMA, isolcpus, фоновая нагрузка — всё влияет. Для
  стабильных чисел: `taskset -c 0,1 make bench`, `cpupower frequency-set
  -g performance`.
- **Два размера — не континуум**: между 256 B и 64 KB есть переломная
  точка (~4–8 KB), где zc начинает окупаться. Для production-решения
  измеряйте на реальных размерах.
- **Один producer, один consumer**. Реальная нагрузка — N×M с миксом
  размеров, там соотношения другие.
- **Go-overhead на cgo**. Для mq/shm+sem Go делает cgo-вызов на каждую
  операцию; в чистом C vs C shm+sem ушёл бы ещё дальше.

## Чего намеренно нет

- **`io_uring`** — async IO нового поколения; не "транспорт данных"
  в классическом смысле.
- **`eventfd`, raw futex** — синхронизация, не сам обмен.
- **Netlink** — kernel↔userspace, а не процесс↔процесс.
- **`taskset`/pinning в harness'е** — числа остаются честные для
  обычного Linux-хоста, а не оптимизированные для рекорда.
