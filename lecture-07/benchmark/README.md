# benchmark — сравнение IPC-транспортов

Ping-pong round-trip между Go-клиентом и C-peer'ом. Каждая итерация —
обмен `MSG_SIZE = 256` байт в обе стороны. Каждый транспорт гоняется
`REPEATS` раз, в таблицу попадают min/median/max ns/round-trip.

Имена транспортов совпадают с `../ipc/01_..07_` для сквозной навигации.

## Запуск

```bash
# Длинный дефолт (~10 минут): REPEATS=50, ITERS=500_000
make bench

# Короткий прогон для разработки/CI (~10 секунд): REPEATS=3, ITERS=100_000
make bench-quick

# Любая промежуточная точка через env:
BENCH_REPEATS=20 BENCH_ITERS=200000 make bench
```

Перед прогоном Makefile собирает C-пиры (`peer_pipe`, `peer_fifo`, …).
Harness: warmup + `REPEATS` измерений на транспорт, сортировка по
медиане (быстрые сверху).

## Типичные цифры (Linux 6.17, i9, governor=performance)

Полный прогон (`make bench` с дефолтами: REPEATS=50, ITERS=500_000).
Суммарное время — 11 мин 26 сек:

```
transport   runs    msgs/sec*    min ns/rt median ns/rt    max ns/rt
------------------------------------------------------------------------
07_shm+sem    50       318779         2913         3136         3505
02_fifo       50       287820         3108         3474         3723
05_mq         50       284050         3168         3520         3844
01_pipe       50       261043         3450         3830         4053
04_uds        50       240379         3891         4160         4401
03_tcp        50       113893         8452         8780         9226
```

`msgs/sec*` посчитан от медианы round-trip'а. С 50-кратным повтором
разброс min/max в пределах ±10%; читать как порядок величин.

## Что видно

- **`07_shm+sem` самый быстрый** (~319k msgs/s): буфер в shared memory,
  syscall'ы только у семафоров (futex wait/wake по одному на round-trip).
  Данные не копируются через kernel — только cache-line traffic между
  CPU-ядрами.
- **`02_fifo`, `05_mq`, `01_pipe`, `04_uds`** — плотной группой ~240–290k
  msgs/s. Все копируют данные через kernel-буфер; разница в метаданных:
  fifo — это pipe с именем в VFS (идентичны по механике), mq оптимизирован
  под сообщения фиксированного размера, uds добавляет sockaddr/ACL-проверки.
- **`03_tcp` на loopback в 2–3× медленнее** (~114k msgs/s): `127.0.0.1`
  не короткий путь — netfilter, qdisc, TCP state machine, checksums.
  `TCP_NODELAY` включён с обеих сторон (без него Nagle добавил бы ~40ms
  на каждый round-trip).

## Почему `06_shm` (raw busy-wait) не в таблице

`ipc/06_shmem/` демонстрирует "самый быстрый IPC без синхронизации":
shmem + volatile-флаги. На коротких прогонах показывает ~3M msgs/s,
на порядок быстрее `shm+sem`. Но на 50×500k прогонах один раз из 50
подвисает из-за memory-ordering между `atomic.Int32` (Go) и
`_Atomic int32_t` (C): оба процесса считают флаг "видимым", но
busy-wait в каком-то окне промахивается. Для бенча важнее
надёжность, поэтому `06_shm` не включён — смотрите standalone-демо
в `../ipc/06_shmem/`.

## Про методологию

- **Warmup**: перед измерениями один прогон выкидываем. Без него первый
  run всегда медленнее из-за холодных кешей, lazy-page-fault'ов, page-table
  population, ядерных буферов.
- **Медиана, не среднее**: выбросы (планировщик увёл нас на busy CPU,
  GC-пауза, preemption) портят mean. Median устойчива.
- **TCP на port=0**: каждый run получает свежий ephemeral-порт от ядра,
  иначе `TIME_WAIT` от предыдущего run'а заставил бы bind ждать 60 сек.
- **Честное копирование в shm+sem**: peer делает `memcpy` в локальный
  буфер и обратно — как реальный консьюмер, а не только `sem_wait/post`.

## Ограничения (не продакшн-бенч)

- CPU governor, NUMA, isolcpus, загрузка — всё влияет. Повторять на
  "холодной" машине с фиксированной частотой; в идеале — `taskset -c 0,1
  make bench` для прибивания к соседним ядрам одного CCX.
- **MSG_SIZE=256 байт** — сравнительно маленькое. При 64K+ shmem уйдёт
  ещё сильнее вперёд, а pipe упрётся в kernel-буфер (~64K — дальше
  write блокируется).
- **Один producer, один consumer**. Реальная нагрузка — N×M с миксом
  размеров, и соотношения там другие.
- **Go-overhead на cgo**. Для mq/shm+sem Go делает cgo-вызов на каждую
  операцию; в чистом C vs C shm+sem ушёл бы ещё дальше.

## Чего намеренно нет

- `SCM_RIGHTS` + `memfd_create` (передача fd через UDS) — отдельная
  тема, не "транспорт данных".
- `eventfd`, raw futex — синхронизация, не сам обмен.
- `io_uring` — следующее поколение async IO, уже не про IPC.
- `taskset`/pinning в harness'е — числа остаются "честные" для среднего
  домашнего/офисного Linux, а не оптимизированные для рекорда.
