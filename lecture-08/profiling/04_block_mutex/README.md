# 04_block_mutex — block и mutex профили

Когда у вас contention, CPU-профиль ничего не покажет: goroutine же не
горит на CPU, она просто **ждёт**. Для этого есть два специальных профиля.

## Запуск

```bash
make run
```

20 goroutine по очереди заходят в критическую секцию, где каждая спит
1 мс — итого 200 мс "полезной работы", но всё это serialized на одном
мьютексе.

## В чём разница

| Профиль | Что считает | Когда читать |
|---|---|---|
| **block** | сколько времени goroutine **ждали** перед Lock() | "почему запросы встают в очередь" |
| **mutex** | сколько времени **удерживался** Lock() (блокируя других) | "какая критическая секция самая долгая" |

Обе метрики дают одно и то же число (сумма ожиданий = сумма удержаний
× (N-1)), но стек другой:

- в **block** стек — это код, **ждущий** Lock (caller → Lock).
- в **mutex** стек — это код **владельца** мьютекса на момент Unlock.

Для оптимизации обычно полезнее **mutex**: он показывает, что именно
держать мьютекс не надо было.

## Включение

Оба профиля **по умолчанию выключены** (overhead). Включаются в рантайме:

```go
runtime.SetBlockProfileRate(1)       // семплируем все события
runtime.SetMutexProfileFraction(1)   // каждое 1-е событие
```

`rate` для block — наносекунды между семплами (1 = всё подряд,
100000 = только события дольше 100 мкс).

В `net/http/pprof` это **не включено по умолчанию** — надо руками
дёрнуть в `init`:
```go
runtime.SetBlockProfileRate(1)
runtime.SetMutexProfileFraction(1)
```

## Анализ

```bash
go tool pprof -top block.pprof
go tool pprof -top mutex.pprof
go tool pprof -http=: mutex.pprof     # flamegraph
```

## Связь с channels

Блокировка на receive/send из канала **тоже** попадает в block-профиль
(под стеком `runtime.chansend1` / `runtime.chanrecv1`). Если у вас
producer-consumer тормозит — смотрите сюда.

## Артефакты

- `block.txt` — топ по времени ожидания.
- `mutex.txt` — топ по времени удержания.

В обоих сверху — наша `main.main.func1` с `time.Sleep` в секции.
