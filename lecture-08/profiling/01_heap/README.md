# 01_heap — heap-профиль и четыре среза одних аллокаций

Программа аллоцирует память в **трёх местах с разной судьбой**, чтобы
на одном профиле сразу было видно, чем отличаются `alloc_*` и
`inuse_*`. Все три функции помечены `//go:noinline` — иначе компилятор
склеит их с `main`, и в pprof останется один узел.

| Функция | Что делает | Размер | Кол-во | Судьба |
|---|---|---|---|---|
| `leakBig` | кладёт буфер в глобальный `leakStore` | 64 KiB | 1000 | живут вечно |
| `churnSmall` | пишет в глобал `sink` (escape), на след. итерации старый умирает | 4 KiB | 100 000 | мертвы |
| `cacheHalf` | кладёт в `cacheStore`, потом половину обнуляем | 16 KiB | 4000 | живёт половина |

Ожидаемая картина:

|  | inuse_space | alloc_space | inuse_objects | alloc_objects |
|---|---|---|---|---|
| `leakBig`    | ~64 MiB | ~64 MiB  | ~1000 | ~1000  |
| `churnSmall` | **~0**  | ~400 MiB | **~0**| ~100k  |
| `cacheHalf`  | ~32 MiB | ~64 MiB  | ~2000 | ~4000  |

`churnSmall` отсутствует в `inuse_*` (GC всё съел), но доминирует в
`alloc_*` — это и есть «давление на GC без утечки».

## Запуск

```bash
make run     # бинарник + 4 среза top-10
make flame   # интерактивный flamegraph в браузере
```

## Четыре `sample_index` — что считают и зачем

| Индекс | Что считает | Когда смотреть |
|---|---|---|
| `alloc_objects` | объектов выделено за всё время | кто давит на GC частотой (мелкие сериализаторы, boxing) |
| `alloc_space`   | байт выделено за всё время       | кто давит на GC объёмом (большие буферы, рендер) |
| `inuse_objects` | живых объектов сейчас            | фрагментация, кеши с мелкими записями |
| `inuse_space`   | живых байт сейчас                | **утечки**, размер кешей |

`alloc_*` растёт **монотонно** — это историческая аллокация. Полезно
искать «какая функция аллоцирует больше всех», даже если эта память
уже собрана GC.

`inuse_*` — **текущий живой heap**. Именно здесь прячется утечка:
если за час `inuse_space` не падает и только растёт — это утечка.

Главное правило диагностики: **`alloc_* − inuse_* = работа GC`**.
Если функция жирная в `alloc_*`, но тощая в `inuse_*` — это **не утечка**,
это горячая аллокация. Лечится через `sync.Pool` / pre-sizing.

## Колонки `top` — что значит каждая

```
      flat  flat%   sum%        cum   cum%
   55.85MB 66.68% 66.68%    55.85MB 66.68%  main.leakBig
   25.90MB 30.92% 97.60%    25.90MB 30.92%  main.cacheHalf
```

- **flat** — ресурс, потраченный *в самой этой функции*, без вложенных.
- **flat%** — доля `flat` от тотала в шапке.
- **sum%** — кумулятивная сумма колонки `flat%` сверху вниз. Бегущая,
  отношения к графу вызовов не имеет — нужна, чтобы быстро увидеть
  «топ-2 дают 97%».
- **cum** — `flat` плюс всё, что эта функция вызвала.
- **cum%** — доля `cum` от тотала.

Правило: `flat ≤ cum`. Если `flat == cum` — это **лист** в графе вызовов.
Если `flat == 0` — функция сама не аллоцирует, она «по пути».

## Почему `runtime.GC()` перед дампом

`pprof.WriteHeapProfile` снимает текущий снапшот, но Go-рантайм не
обязан помечать «недавно умершие» объекты как мёртвые до следующего
цикла GC. На горячем сервисе без `runtime.GC()` `inuse_*` будет
завышен — увидите утечку, которой нет. В нашем примере без `GC()`
вы бы увидели в `inuse` ещё и `churnSmall` на ~400 MiB.

## Семплинг — что **не** надёжно

- Дефолт `runtime.MemProfileRate` = **512 KiB**: одна аллокация
  семплируется на каждые 512 KiB байт. Поэтому `leakBig` показывает
  893 объекта вместо ровных 1000 — это шум семплинга, не баг.
- `pprof` экстраполирует тотал из семплов, поэтому числа округлые
  и слегка плавают между запусками.
- Стек берётся **только Go-шный**. CGo-аллокации (`C.malloc`) heap-профиль
  не видит вообще.

## Команды

```bash
go tool pprof -top -sample_index=inuse_space heap.pprof
go tool pprof -top -sample_index=alloc_space heap.pprof

# Интерактивно:
go tool pprof heap.pprof
(pprof) top
(pprof) top10 -cum
(pprof) list main.leakBig    # исходник с heat-map по строкам
(pprof) peek main.cacheHalf  # вызывающие/вызываемые
(pprof) sample_index alloc_space
(pprof) web                  # SVG-граф (graphviz)

# HTTP UI с flamegraph:
make flame
```

## Diff двух снимков — главный инструмент в проде

```bash
curl localhost:6060/debug/pprof/heap > t0.pprof
# ...нагрузка 10 минут...
curl localhost:6060/debug/pprof/heap > t1.pprof
go tool pprof -base t0.pprof t1.pprof
```

`-base` вычитает t0 из t1: `top` покажет **только то, что выросло**.
Это убирает шум легитимной памяти (рантайм, кеши, пулы) и оставляет
дельту. Если `main.leakBig` весит +100 MiB в дельте — поймали.
