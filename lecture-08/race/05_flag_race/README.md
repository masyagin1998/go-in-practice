# 05_flag_race — "безобидной гонки не бывает"

Воркер крутит `for !done`, другая goroutine через 100 мс ставит
`done = true`. Без синхронизации это data race. По memory model Go
у нас нет гарантии, что писатель и читатель когда-либо согласуются.

## Запуск

```bash
go run .          # на современном Go обычно "успели"
go run -race .    # WARNING: DATA RACE, exit 66
```

## Почему без `-race` не виснет (на современном Go)

Раньше популярный аргумент звучал так: "компилятор имеет право вынести
load `done` из цикла (loop-invariant code motion) — `for !done`
превратится в `for true`". Формально по memory model это правда.

На практике на Go 1.14+ это **почти никогда не происходит**:

- В каждой итерации цикла есть проверка async-preemption (вызов в
  рантайм для возможной приостановки G). Это side-effect, и SSA-бэкенд
  не выносит load глобала через него.
- Глобал `done` лежит в bss-сегменте, и компилятор по умолчанию
  консервативен с глобалами без `//go:nosplit` и т.п.

Поэтому в нашей демке воркер обычно крутит ~800M спинов за 100 мс и
честно выходит. Это **везение, а не корректность**.

## Почему это всё равно сломанный код

Формально программа невалидна. По memory model Go без явной
синхронизации:

> Записи из одной goroutine не обязаны становиться видны в другой —
> ни сейчас, ни когда-либо.

Компилятор может закешировать в регистре, вынести load из цикла,
переупорядочить независимые записи. CPU тоже. На x86 модель сильнее,
чем кажется глазам; на ARM/POWER — намного слабее, и тот же код может
там залипнуть.

`-race` ловит это **детерминированно**, по vector-clock'ам, не глядя
на то, "повезло" ли вам сегодня с компилятором.

## Фикс — в `06_atomic_fix/`

`-race` обязателен в CI: 5–10× замедление по CPU и +5–10× по памяти,
но только в тестовой сборке; в прод не льётся.
