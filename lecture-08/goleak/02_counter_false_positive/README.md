# 02_counter_false_positive — почему счётчика недостаточно

`runtime.NumGoroutine()` — снимок счётчика. Утечка ≠ цифра выросла;
цифра выросла ≠ утечка. Демонстрируем два типичных false-positive.

## Запуск

```bash
go run .
```

Ожидаемый вывод:
```
ранний снимок (через 20мс):  +10 goroutine
→ выглядит как 10 утёкших — но они просто не успели завершиться
поздний снимок (через 320мс): +0 goroutine
→ утечки нет, все вышли сами

после http.Get: +2 goroutine
→ это keep-alive readLoop/writeLoop, не наша утечка
```

## Кейс 1. "Ещё не успели выйти"

Goroutine честно завершается, но не мгновенно. Если сэмплировать
сразу после `cancel()` / `wg.Done()` — увидим её и решим, что утекла.

В реальности так выглядит graceful shutdown: дренаж очереди, закрытие
соединений, flush буферов. Время выхода 50мс…2с — в пределах нормы.

## Кейс 2. "Это вообще не наши goroutine"

`net/http` при первом запросе к хосту поднимает per-connection
`readLoop` + `writeLoop` для keep-alive pool. Они переживают сам
`Get` — ровно для того, чтобы следующий запрос на этот же хост был
быстрее. Это не утечка, это дизайн.

Похожий шум:
- `database/sql` connection pool — N idle-коннекций × N goroutine.
- `runtime/pprof.StartCPUProfile` — фоновый sample reader.
- DNS resolver при `LookupHost` — может оставить goroutine в ожидании
  ответа cgo-резолвера.

Пакет, который вы просто **импортировали**, может стартовать
goroutine в `init()` — и она не "ваша", даже если сидит в вашем
снимке.

## Чем заменить счётчик

| Подход | Что ловит | Чего не ловит |
|---|---|---|
| `NumGoroutine()` снимок | grossly заметный устойчивый рост | точечные утечки, шумит |
| `NumGoroutine()` тренд (метрика) | устойчивый рост во времени | точечные утечки в одном тесте |
| `goleak.VerifyNone` (см. `../03_`) | точечные, со стеками, с фильтрами | многопроцессные утечки |
| `pprof.Lookup("goroutine")` | абсолютно всё | без diff'а — много шума |

## Правило

Счётчик — для метрики в проде ("Goroutines/sec за неделю → растут?").
Для тестов — `goleak`: он сверяет goroutine **по стектрейсам** и умеет
игнорировать известный фон через `IgnoreTopFunction`/`IgnoreCurrent`.
