# 01_leak_basic — утечка видна цифрой

Самый простой индикатор: `runtime.NumGoroutine()` вырос — что-то утекло.

Здесь стартуем 10 заведомо-утекающих goroutine: 5 блокируются на send
в unbuffered chan без читателя, 5 — на recv из канала без писателя.
В обоих случаях канал и goroutine держат друг друга → GC не соберёт.

## Запуск

```bash
go run .
```

## Чего не хватает

Цифра растёт — но **где** именно висят goroutine, неизвестно. В реальном
сервисе утечка приходит из десятков библиотек, и без стектрейсов искать
её невозможно.

Два пути дальше:

1. `runtime/pprof.Lookup("goroutine").WriteTo(os.Stdout, 2)` — снимок
   стеков всех живых goroutine. Шумно: видим всё, а не diff.
2. `go.uber.org/goleak` — diff "до/после" с фильтрацией шума, оформлен
   как падающий тест. См. `../03_goleak_test/`.

## Частые паттерны утечек

- **Забытый `<-ch`** или `ch <- v` — этот пример.
- **Забытый `ctx.Done()`** в for-select — см. `../04_context_leak/`.
- **`time.After` в цикле** — каждый вызов создаёт таймер; правильный
  путь — `time.NewTimer` + `Reset`.
- **`http.Response.Body` без `Close()`** — keep-alive readLoop держит
  goroutine на соединение.
- **`wg.Add()` после `wg.Wait()`** — гонка + пропущенный счётчик.

## Утечка goroutine ≠ только memory leak

Пока goroutine жива — её стек жив (8 KiB + всё, на что она ссылается),
а с ней живут OS-ресурсы (сокеты, файлы). Утечка goroutine = утечка
памяти + утечка fd.
