# Атомарность на уровне процессора: результаты

## Три демо

| # | Демо                  | Аналог в C                                                 | offset | sizeof | Torn read?                 |
|---|-----------------------|------------------------------------------------------------|--------|--------|----------------------------|
| 1 | expected-atomic       | `struct { int32_t shit; int64_t value; }`                  | 8      | 16     | Нет (0 из 740M+ итераций)  |
| 2 | unexpected-atomic     | `__packed__ struct { int32_t shit; int64_t value; }`       | 4      | 12     | Нет (0 из 840M+ итераций)  |
| 3 | expected-non-atomic   | `__packed__ __aligned__(64) struct { char[60]; int64_t; }` | 60     | 68     | Да (20 из 81 итерации!)    |

## Что происходит

На x86-64 выровненный 8-байтовый load/store атомарен (Intel SDM §9.1.1).
Но «выровненный» — это не про кратность 8. Это про **кэш-линию**:

```
Кэш-линия (64 байта):
┌──────────────────────────────────────────────────────────────────┐
│ 0          4          8                                    60   │ 63
└──────────────────────────────────────────────────────────────────┘

expected-atomic:     [████████]          value @ offset 8  → внутри линии ✓
unexpected-atomic:       [████████]      value @ offset 4  → внутри линии ✓
expected-non-atomic:                                       [████│████]
                                                           60   64
                                                           ↑ разрыв кэш-линии!
```

- **expected-atomic** — Go выравнивает `int64` по 8 байтам, добавляя padding.
  value на offset 8, целиком в одной линии. Torn read невозможен.

- **unexpected-atomic** — эмулируем `__packed__` (без padding), value на offset 4.
  Невыровнен по 8, но всё ещё внутри одной кэш-линии.
  **На x86-64 torn read НЕ возникает** — CPU гарантирует атомарность
  для доступов внутри одной линии.

- **expected-non-atomic** — value на offset 60, байты [60..63] в одной линии,
  [64..67] в следующей. CPU обращается к двум линиям **неатомарно** →
  torn read возникает на каждой четвёртой итерации.

## Подводные камни Go

### Dead store elimination

Go-компилятор оптимизирует:
```go
*ptr = ValA
*ptr = ValB
// → только *ptr = ValB (ValA — «мёртвый» store)
```

В C это предотвращает `volatile`. В Go аналога нет — используем
`//go:noinline` функции `store64`/`load64`, чтобы компилятор не мог
заглянуть внутрь и выкинуть store.

### Нет `__packed__`

Go всегда выравнивает поля структуры. `struct { int32; int64 }` в Go
занимает 16 байт (с 4 байтами padding), а не 12.
Единственный способ разместить `int64` на произвольном смещении —
арифметика с `unsafe.Pointer`.

### Нет `volatile`

Go не гарантирует, что обращение к переменной превратится в реальный
load/store. Компилятор может кэшировать значение в регистре, переупорядочить
или удалить обращения. `//go:noinline` на store/load — минимальный workaround.
