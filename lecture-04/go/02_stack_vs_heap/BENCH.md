# Stack vs Heap — результаты бенчмарка

## Окружение

- Go 1.26.0, linux/amd64
- GOGC=100 (default), GOMEMLIMIT не задан
- Структура `Data` содержит `[64]int` = 512 байт

## Результаты (10 000 000 итераций)

| Метрика               | Стек          | Куча              |
|-----------------------|---------------|-------------------|
| Время                 | ~30 мс        | ~1200 мс          |
| Аллокаций в куче      | 0             | 10 000 000        |
| Память                | 0 КБ          | ~5 000 000 КБ     |
| Циклов GC             | 0             | ~1700             |
| **Разница**           |               | **~40x медленнее** |

## Почему такая разница

**Стек** — аллокация это просто сдвиг регистра SP (stack pointer).
При выходе из функции SP сдвигается обратно — «освобождение» бесплатно.
GC вообще не задействован.

**Куча** — каждая аллокация это вызов `runtime.mallocgc`:
1. Поиск свободного блока нужного размера в span'ах.
2. Запись в bitmap для GC (какие слова — указатели).
3. Периодический запуск GC (~1700 циклов за 10M аллокаций).
4. GC сканирует все живые объекты (mark phase) и освобождает мёртвые (sweep phase).

## Вывод

Escape analysis в Go — не просто оптимизация, а критическая разница
в производительности. Если структура не убегает из функции — она бесплатна.
Если убегает — платим за `mallocgc` + давление на GC.

Практические правила:
- Возвращайте по значению маленькие структуры (до ~256 байт).
- Избегайте передачи через `interface{}` / `any` без необходимости.
- Проверяйте решения компилятора: `go build -gcflags="-m -l" .`
