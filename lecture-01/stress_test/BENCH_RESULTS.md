# Stress test: 1 000 000 конкурентных единиц, sleep 10 сек

Цель: запустить 1 000 000 параллельных «задач» (горутин / OS-тредов),
каждая из которых просто спит 10 секунд. Показать разницу в стоимости
горутины vs OS-треда.

## Окружение

- CPU: 14 ядер (28 потоков)
- RAM: 32 GB
- OS: Linux 6.17.0, Ubuntu
- Go: go1.24
- C: gcc, pthreads, стек по умолчанию (~8 MB на тред)

## Результаты

| Метрика                 | Go (горутины)        | C (pthreads)               |
|-------------------------|----------------------|-----------------------------|
| Цель                    | 1 000 000            | 1 000 000                   |
| Фактически создано      | **1 000 000**        | **75 969** (EAGAIN)         |
| Время запуска           | 1.3 сек              | 0.7 сек                     |
| Общее время выполнения  | 11.3 сек             | 10.7 сек                    |
| OS-тредов               | **30**               | **75 970**                  |
| VmSize (виртуальная)    | 3 856 MB             | **608 087 MB**              |
| VmRSS (физическая)      | 2 634 MB             | 616 MB                      |
| RAM на единицу (RSS)    | ~2.7 KB / горутина   | ~8.3 KB / тред              |
| Стек на единицу         | 2-8 KB (растущий)    | 8 MB (по умолчанию, lazy)   |

## VmSize vs VmRSS: почему 622 GB виртуальной — и всего 0.6 GB физической

76K тредов × 8 MB стек = 608 GB виртуальной памяти. Но `mmap(MAP_ANONYMOUS)`
не выделяет физические страницы — они создаются **лениво**, при первом
обращении (page fault). Спящий тред трогает ~1 страницу (4 KB) из 8 MB стека:

```
thread_func()  — 1 фрейм, ~16 байт
sleep()        — обёртка libc, ~32 байта
→ syscall      — уход в kernelspace, стек userspace больше не трогается
```

Поэтому RSS = 76K × ~8 KB ≈ 616 MB, а не 608 GB.

Go наоборот: стеки горутин аллоцируются из Go heap (реальная память),
а не через lazy mmap. Поэтому VmRSS у Go выше (2 634 MB), несмотря на
меньшие стеки.

## Почему C не смог создать 1 000 000 тредов

`pthread_create` вернул `EAGAIN` на ~76K тредов. Причины:

1. **`/proc/sys/kernel/threads-max`** = 510 232 — глобальный лимит с учётом
   всех процессов в системе. ~76K — то, что осталось свободным.
2. **Каждый pthread = `clone()` syscall** → запись в task_struct ядра (~5-10 KB),
   PID, стек в kernelspace (16 KB), запись в scheduler CFS tree.
3. **`vm.max_map_count`** = 1 048 576 — каждый тред создаёт 2 mmap-области
   (стек + guard page), 76K тредов ≈ 152K mmap'ов.

Для запуска 250K+ тредов нужно:
```bash
sudo sysctl -w kernel.threads-max=2100000
sudo sysctl -w kernel.pid_max=2100000
sudo sysctl -w vm.max_map_count=4200000
```

## Почему Go справился с 1 000 000

Горутина — **не OS-тред**. Это структура `runtime.g` (~400 байт) + маленький стек
(начальный 2 KB, растёт по требованию до 1 GB). Все 1 000 000 горутин
мультиплексируются на **30 OS-тредов** (M:N scheduling).

- `time.Sleep` не блокирует OS-тред — рантайм ставит таймер и переключает
  горутину. Один тред обслуживает тысячи «спящих» горутин.
- Нет `clone()` на каждую горутину — нет давления на ядро.
- Нет mmap на каждую горутину — стеки аллоцируются из heap.

## Выводы

- **Горутины в ~13x дешевле по созданию**: Go создал 1M за 1.3 сек, C —
  76K за 0.7 сек (если пересчитать, C тратит ~9 мкс/тред, Go — ~1.3 мкс/горутину).
- **OS-треды упираются в лимиты ядра** задолго до исчерпания RAM.
  76K тредов заняли 608 GB виртуальной памяти — и это при том, что
  физически используется только 616 MB.
- **Go использует больше RSS** (2 634 MB vs 616 MB), но создал в 13x больше
  единиц конкурентности. В пересчёте на единицу Go втрое дешевле.
- **30 OS-тредов вместо 76K** — Go не масштабирует количество тредов
  с количеством задач. Это принципиальное архитектурное отличие M:N
  планировщика от модели 1:1.
